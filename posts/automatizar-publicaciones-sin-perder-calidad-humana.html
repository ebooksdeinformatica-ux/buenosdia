Automatizar no es publicar cien veces el mismo cadáver

La automatización buena arma el taller. La mala llena el blog de textos que parecen vivos hasta que los leés.

Por ASPF · Actualizado 12/06/2026 · 8–11 min
WhatsAppFacebookX

Este artículo no debería existir como teoría. Debería existir como cicatriz.

Porque un blog puede llenarse rápido. Muy rápido. Se pueden generar títulos, descriptions, secciones, preguntas frecuentes, enlaces, imágenes y hasta un sitemap entero. Desde afuera parece trabajo. Desde adentro, si el contenido nació sin mirada, es una fábrica de páginas vacías.

Eso fue exactamente lo que había que corregir: textos con distinto título, pero con la misma respiración. Distinta puerta, misma habitación.

La parte que sí conviene automatizar

El sitio necesita máquinas para no volverse inmanejable. No tiene sentido escribir a mano cada canonical, cada tarjeta social, cada entrada del sitemap o cada listado de categoría.

Ahí GitHub, scripts y workflows son aliados. Ordenan, comparan, frenan errores, actualizan catálogo y dejan registro. Ese tipo de automatización no le roba alma al blog; le pone piso.

El problema aparece cuando esa misma fuerza se usa para fabricar la voz.

La parte que no se delega

Una máquina puede decir si falta una meta description. No puede saber si el texto tiene una herida real, una observación propia o una frase que solo podría haber escrito alguien que estuvo ahí.

Puede detectar repetición literal. No siempre detecta una cadencia clonada: apertura prolija, tres párrafos de contexto, lista de pasos, FAQ genérica, cierre motivacional. Todo correcto. Todo muerto.

El freno nuevo

Desde la actualización del modelo, la pregunta cambió.

Antes: ¿pasó auditoría?

Ahora: ¿si le cambio el título, el cuerpo sigue sirviendo para otro artículo?

Si la respuesta es sí, el texto vuelve al taller.

GitHub como taller, no como imprenta

La rama de trabajo tiene que ser incómoda. Tiene que mostrar qué cambió, qué falló, qué se reescribió y qué todavía huele a molde. La PR no es una cinta transportadora hacia producción. Es una mesa de disección.

Por eso usar GitHub como taller de proyectos tiene más sentido que usarlo solo como botón de publicar.

Una automatización digna hace tres cosas

Primero, guarda estructura técnica. Segundo, obliga a revisar. Tercero, deja evidencia de por qué algo no se publica.

Lo que no debe hacer es esconder mediocridad detrás de un check verde.

Control editorial real

Para este blog, una publicación nueva tiene que pasar por una lectura simple:

¿Abre distinto? ¿Trae un ejemplo propio? ¿Tiene una razón para existir? ¿Los enlaces son naturales? ¿La fuente responde al tema? ¿El cierre pertenece a este texto y no a otros veinte?

En usar IA para mejorar textos sin perder tu voz está el mismo problema desde el lado de la escritura: la herramienta ayuda, pero no puede quedarse con el volante.

Preguntas que frenan la publicación

¿Qué automatizo sin miedo?

Lo técnico repetible: sitemap, catálogo, validaciones, enlaces comunes, build, formato y chequeos.

¿Qué no debería salir automático?

La voz, el enfoque, la experiencia, el ejemplo central y la decisión de publicar.

¿Un workflow verde alcanza?

No. El verde técnico solo dice que la página funciona. No dice que el artículo merezca existir.
