Automatizar sirve cuando elimina una repetición, no cuando esconde el desorden

La mejor primera automatización no es la más impresionante. Es la que podés explicar, probar y recuperar si algo sale mal.

Por ASPF · Actualizado 12/06/2026 · 11–14 min
WhatsAppFacebookX

Automatizar tiene una fama extraña. Parece que todo proceso mejora apenas le agregamos un script, una inteligencia artificial o una plataforma que conecta botones. Pero una automatización también puede repetir un error cientos de veces, borrar archivos más rápido o dejarte dependiendo de un sistema que nadie entiende.

La primera regla es incómoda: no automatices un proceso que todavía no sabés hacer manualmente. Si cada vez cambia la entrada, la decisión y el resultado, no existe una tarea estable. Existe una situación que todavía necesita criterio.

Elegí una repetición que puedas describir

Una buena candidata aparece con frecuencia, sigue pasos parecidos y produce un resultado verificable. Por ejemplo:

  • Renombrar imágenes con una convención.
  • Leer un CSV y separar registros válidos de inválidos.
  • Generar páginas HTML desde datos estructurados.
  • Comprobar que una lista de enlaces no tenga duplicados.
  • Copiar archivos terminados a una carpeta de respaldo.
  • Actualizar un sitemap después de publicar.

“Gestionar todo mi negocio” no es una automatización. Es un conjunto enorme de decisiones. “Crear una copia fechada de los archivos publicados” sí puede transformarse en un proceso pequeño.

Escribí el proceso antes del código

Ejecutá una vez la tarea y anotá:

  1. Qué recibe.
  2. Qué valida.
  3. Qué transforma.
  4. Qué produce.
  5. Dónde guarda el resultado.
  6. Qué debe hacer si falta algo.

Ese documento vale incluso si después usás una herramienta sin código. Te permite distinguir la lógica del botón que la ejecuta.

Un flujo para optimizar imágenes podría decir: recibir archivos originales, aceptar únicamente extensiones definidas, crear una copia, redimensionar, convertir, guardar en otra carpeta y producir un informe. Si el proceso solo dice “achicar imágenes”, todavía deja demasiadas decisiones escondidas.

Entrada, salida y registro

Toda automatización debería tener límites visibles. La entrada es el material que puede tocar. La salida es lo que debe crear. El registro explica qué ocurrió.

Sin registro, un proceso que termina en silencio obliga a revisar todo manualmente. Un informe puede incluir archivos procesados, ignorados, fallidos y duración. No hace falta un sistema enorme: un archivo de texto o CSV puede alcanzar.

La biblioteca estándar de Python incluye pathlib, que representa rutas de archivos con objetos adecuados para distintos sistemas operativos. Permite recorrer carpetas, consultar tipos, leer, escribir, copiar o mover. El módulo csv implementa lectura y escritura de datos tabulares con variantes del formato. Son herramientas suficientes para muchas automatizaciones pequeñas sin instalar docenas de dependencias.

Trabajá siempre sobre una copia al principio

Una automatización nueva no debería tocar el único conjunto de datos real. Creá una carpeta de prueba con muestras pequeñas, casos normales y casos problemáticos.

Incluí nombres con espacios, campos vacíos, duplicados, caracteres acentuados y archivos inesperados. El objetivo no es demostrar que funciona en el ejemplo ideal. Es descubrir qué hace cuando la realidad no respeta tu imaginación.

Antes de permitir borrados o sobrescrituras, usá un modo de simulación. El programa puede mostrar qué haría sin hacerlo. Después comparás la lista y recién habilitás el cambio.

Una sola responsabilidad por herramienta

Cuando un script descarga, edita, publica, envía correos y borra archivos, cualquier falla se vuelve difícil de aislar. Es mejor separar etapas que puedan verificarse.

Por ejemplo:

  1. Validar datos.
  2. Generar archivos.
  3. Ejecutar auditorías.
  4. Publicar únicamente si todo pasó.

Así trabaja el flujo nuevo de buenosdia.com: el contenido se reescribe, el catálogo se actualiza, la auditoría editorial revisa frases repetidas y otra auditoría controla canonical, enlaces, datos estructurados y Statcounter. Si un control falla, no se publica.

Automatizar con GitHub Actions

GitHub Actions permite crear flujos que se ejecutan cuando ocurre un evento, como un push o un pull request. Un workflow contiene uno o más jobs; cada job corre en un runner y contiene pasos que ejecutan scripts o acciones reutilizables.

Para un blog estático puede servir para:

  • Validar HTML.
  • Comprobar enlaces internos.
  • Generar sitemap.
  • Actualizar listados.
  • Rechazar contenido con frases prohibidas.
  • Publicar solo después de aprobar controles.

No conviene meter secretos directamente en el archivo del workflow. GitHub dispone de secretos y permisos configurables. También hay que limitar qué puede escribir cada proceso.

El error debe detener o aislar, no desaparecer

Un try/except que atrapa todo y continúa puede ocultar que la mitad de los archivos falló. Manejar errores significa decidir cuáles son recuperables, cuáles requieren detenerse y qué información necesita la persona que revisará.

Un mensaje útil dice qué archivo, qué operación y qué excepción ocurrió. “Error” no alcanza.

También definí qué pasa si el proceso se interrumpe a mitad. ¿Puede reanudarse? ¿Reprocesa archivos? ¿Duplica resultados? Las respuestas importan más cuando el volumen crece.

Medí el ahorro completo

Una automatización que ahorra diez minutos pero necesita dos horas de mantenimiento semanal no ahorra. Medí:

  • Tiempo manual anterior.
  • Tiempo de ejecución.
  • Tiempo de preparación y revisión.
  • Errores evitados o introducidos.
  • Frecuencia de uso.
  • Costo de herramientas.
  • Tiempo necesario para recuperar una falla.

La primera versión puede no ahorrar. Está pagando aprendizaje. Pero después de varias ejecuciones debería aparecer una diferencia.

Cuándo no automatizar

No automatices una tarea que ocurre una sola vez y tarda menos que construir el sistema. Tampoco una decisión que requiere interpretar situaciones humanas, legales, médicas o financieras sin reglas claras.

Si el proceso cambia cada semana, primero estabilizalo. Si una equivocación puede borrar datos o enviar información sensible, agregá aprobación humana antes del paso irreversible.

Un primer proyecto razonable

Elegí una carpeta con copias de imágenes y construí un script que liste nombre, tamaño y extensión en un CSV. No modifica nada. Solo observa y registra.

Después agregá una regla: marcar archivos mayores a cierto tamaño. Luego podés copiar los seleccionados a otra carpeta. Cada etapa tiene una salida comprobable y no necesita comenzar destruyendo nada.

Cuando eso funcione, guardá el script, agregá instrucciones y usá Git. Una automatización sin documentación vuelve a convertirse en trabajo manual el día que olvidás cómo se ejecuta.

Preguntas frecuentes

¿Necesito saber programar para automatizar?

No para todas las tareas. Existen herramientas visuales y funciones integradas. Pero entender entradas, reglas, errores y salidas sigue siendo necesario.

¿Python sirve para automatizar archivos en Windows?

Sí. Su biblioteca estándar incluye herramientas para rutas, archivos, CSV y procesos. La compatibilidad exacta depende de la operación y del entorno.

¿Cuándo conviene usar GitHub Actions?

Cuando la tarea pertenece al repositorio y debe ejecutarse ante eventos como cambios o pull requests. Para acciones locales sin repositorio puede convenir otro sistema.

¿La IA puede escribir la automatización?

Puede ayudar, pero el código debe revisarse y probarse con datos no críticos. Nunca entregues credenciales ni ejecutes comandos que no entendés.
