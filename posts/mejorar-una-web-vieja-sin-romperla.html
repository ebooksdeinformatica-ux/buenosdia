La primera mejora de una web vieja es no romper lo que todavía sirve

Antes de rediseñar, copiá, medí, entendé dependencias y cambiá por capas. La prisa técnica suele salir cara.

Por ASPF · Actualizado 12/06/2026 · 9–12 min
WhatsAppFacebookX

Una web vieja puede dar vergüenza, pero también puede tener algo valioso: URLs conocidas, contenido indexado, enlaces, historial, archivos que todavía cargan y una estructura que alguien, en algún momento, logró poner en marcha.

El error clásico es querer modernizar todo en una tarde. Cambiar diseño, rutas, imágenes, scripts y contenido al mismo tiempo. Si algo se rompe después, no sabés qué lo causó.

Mejorar sin romper exige paciencia técnica. Primero entender, después tocar.

Hacé una copia antes de opinar

Antes de editar, descargá o cloná el sitio. Guardá una versión completa con fecha. Si usa repositorio, creá una rama. Si es un sitio estático, copiá archivos. Si hay base de datos, necesitás un respaldo real y una forma probada de restaurar.

No alcanza con decir “después lo arreglo”. Una web rota bajo presión se vuelve mucho más difícil.

Revisá qué está vivo

La auditoría inicial no tiene que ser perfecta. Necesitás saber:

  • qué páginas reciben visitas;
  • qué URLs están enlazadas;
  • qué formularios o contactos funcionan;
  • qué scripts externos cargan;
  • qué imágenes pesan demasiado;
  • qué contenido está desactualizado;
  • qué errores aparecen en móvil.

Sin ese mapa, podés “mejorar” justo lo que mantenía al sitio funcionando.

No cambies URLs por estética

Una URL vieja puede no ser linda, pero si recibe enlaces o tráfico, cambiarla sin redirección es tirar señales. Si necesitás cambiar rutas, hacelo con redirecciones y comprobación.

Cuando una URL es útil, conservá. Cuando es engañosa, duplicada o imposible, planificá el cambio.

Separá capas

Trabajá por capas: primero seguridad y respaldo, después rendimiento, después contenido, después diseño. No mezcles todo.

Un cambio pequeño puede ser comprimir imágenes. Otro, corregir enlaces. Otro, mejorar titles y descriptions. Otro, adaptar móvil. Cada cambio debe poder probarse.

Usá control de versiones

Git permite registrar qué cambió y volver atrás. Incluso en una web simple, hacer commits pequeños evita perder contexto.

Un mensaje como “optimizar imágenes de portada” vale más que “cambios varios”. Si algo falla, encontrás el punto.

Para ordenar proyecto y versiones, leé crear un sistema de trabajo digital desde cero.

Prioridad: usuario primero

Antes de perseguir efectos visuales, revisá si la persona entiende la página. Qué ofrece, para quién, cómo sigue, cómo contacta y si puede leer desde el teléfono.

Una web antigua con texto claro puede vender mejor que un rediseño moderno que tarda en cargar y es confuso.

Mejoras que suelen rendir

  • Comprimir imágenes grandes.
  • Eliminar scripts que ya no se usan.
  • Corregir enlaces internos rotos.
  • Ordenar encabezados.
  • Agregar canonical correcto.
  • Actualizar datos de contacto.
  • Mejorar vista móvil.
  • Crear sitemap limpio.

Son mejoras menos glamorosas que un rediseño total, pero suelen impactar en experiencia, mantenimiento y SEO.

Probá antes de publicar

Abrí páginas principales en escritorio y móvil. Revisá enlaces, imágenes, formularios y consola del navegador. Si el sitio tiene medición, comprobá que el código siga presente.

Una checklist evita confiar en memoria.

Cuándo conviene rehacer desde cero

Si la tecnología no tiene soporte, la estructura es irrecuperable, no hay manera de mantener el sitio o la seguridad está comprometida, quizá convenga reconstruir. Aun así, conservá URLs valiosas, contenido útil y señales existentes.

Rehacer no significa borrar historia. Significa trasladar lo rescatable a una base más sana.

Plan mínimo

  1. Copiar sitio completo.
  2. Listar páginas importantes.
  3. Medir peso y problemas visibles.
  4. Crear rama de trabajo.
  5. Corregir una capa.
  6. Probar.
  7. Publicar solo si pasa la checklist.

Una web vieja se mejora mejor cuando cada paso puede explicarse y revertirse.

Preguntas frecuentes

¿Conviene rediseñar todo?

No siempre. Primero revisá qué funciona y qué está roto. A veces pequeñas mejoras producen más valor que un rediseño completo.

¿Puedo cambiar todas las URLs?

Solo con plan, redirecciones y revisión. Cambiar URLs útiles sin control puede perder tráfico y enlaces.

¿Qué hago si no hay backup?

Creá uno antes de tocar. Si hay base de datos, necesitás exportación y prueba de restauración.

¿Cuál es la primera mejora?

La que reduce riesgo: copia, inventario, enlaces, imágenes pesadas y contacto visible.
