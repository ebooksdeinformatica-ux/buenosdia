Una web vieja no se arregla entrando con martillo

Primero hay que mirar qué todavía vive ahí: URLs, visitas, textos, enlaces, archivos y pequeñas cosas que sostienen el sitio.

Por ASPF · Actualizado 12/06/2026 · 9–12 min
WhatsAppFacebookX

Una web vieja puede dar vergüenza. Colores de otra época, archivos con nombres raros, imágenes pesadas, HTML mezclado, textos que ya no representan nada. La tentación es borrar todo y empezar limpio.

Pero una web vieja también puede guardar señales que no se ven a simple vista: URLs indexadas, enlaces externos, visitas pequeñas pero constantes, artículos que alguien todavía encuentra, una estructura que Google ya conoce y archivos que, con todos sus defectos, siguen cargando.

Antes de mejorar, conviene bajar la ansiedad.

La copia es el permiso para tocar

La primera mejora no se ve en pantalla. Es tener una copia. Un clon, una rama, un respaldo con fecha, algo que permita volver si el entusiasmo rompe una pieza importante.

Sin copia, cada cambio se vuelve apuesta. Con copia, se vuelve taller.

Una web vieja tiene zonas vivas y zonas muertas

No todo merece conservarse. Pero tampoco todo merece destruirse.

Hay páginas que todavía reciben visitas. Hay enlaces internos que sostienen recorrido. Hay títulos antiguos que, aunque feos, responden búsquedas. Hay archivos que conviene reemplazar. Hay secciones que ya no tienen sentido.

La tarea no es admirar la ruina. Es distinguir.

No cambies rutas por estética

Una URL vieja puede ser incómoda, pero si tiene señales, cambiarla sin plan puede costar caro. Si una ruta debe cambiar, necesita redirección y revisión.

Google documenta las redirecciones porque los cambios de URL no son un detalle visual. Son una mudanza.

Mejorar por capas

Una capa puede ser imágenes. Otra, enlaces rotos. Otra, titles. Otra, móvil. Otra, contenido viejo. Otra, rendimiento.

Si tocás todo a la vez, después no sabés qué ayudó ni qué rompió. El cambio chico parece lento, pero deja huella entendible.

El rediseño no siempre es la prioridad

Muchas webs viejas no necesitan primero verse modernas. Necesitan cargar, explicar, enlazar bien, tener contacto visible, no romper en móvil y no ocultar lo importante debajo de ruido.

Un diseño nuevo sobre una estructura confusa solo maquilla el problema.

GitHub como mesa de trabajo

Cuando el proyecto puede vivir en Git, cada cambio queda registrado. No hace falta dramatizar: un commit por cambio razonable, un mensaje claro y una rama donde probar.

En usar GitHub como taller de proyectos esta idea es central: no publicar desde impulso, sino desde comparación.

La pregunta antes de publicar

Después de cada mejora, abrí la web como visitante. ¿Entiendo dónde estoy? ¿Carga rápido? ¿Puedo seguir leyendo? ¿Los enlaces funcionan? ¿El móvil respira? ¿El cambio conservó lo que valía?

Si no podés responder eso, todavía no publiques.

Preguntas para no destruir valor viejo

¿Qué miro antes de cambiar?

URLs con señales, páginas visitadas, enlaces, contacto, móvil, imágenes pesadas y contenido todavía útil.

¿Cuándo conviene rehacer todo?

Cuando la base ya no se puede mantener o tiene riesgos serios. Aun así, se rescatan rutas y contenido valioso.

¿Qué cambio hago primero?

El que reduce riesgo: copia, inventario, imágenes, enlaces y páginas principales.
