GitHub puede ser tu taller, no solo un depósito de código

Un proyecto necesita versiones, pruebas, notas y una forma de avanzar sin romper lo estable. GitHub puede ordenar ese trabajo.

Por ASPF · Actualizado 12/06/2026 · 9–12 min
WhatsAppFacebookX

GitHub suele parecer una herramienta lejana si uno no viene de programar todos los días. Pero para un proyecto digital puede funcionar como taller: ahí se guardan versiones, se prueban cambios, se documentan decisiones y se publica sin tocar a ciegas lo que ya funciona.

Un taller no es un museo. No todo está perfecto. Hay ramas, pruebas, borradores, errores y mejoras. La diferencia es que cada cosa tiene lugar y se puede recuperar.

Para un blog como buenosdia.com, GitHub permite separar lo estable de lo que está en reparación. Eso evita el desastre de cambiar cientos de archivos sin control.

Repositorio: la mesa de trabajo

Un repositorio contiene archivos y su historial. No es solo una carpeta online. Cada cambio puede quedar registrado con fecha, autor y mensaje.

Eso sirve para HTML, CSS, scripts, documentación, notas de avance y configuraciones. Cuando algo se rompe, el historial ayuda a entender qué cambió.

Commits pequeños

Un commit debería contar una unidad de avance. “Actualizar todo” no ayuda. “Reescribir artículo sobre web estática” sí.

Los commits pequeños permiten revisar, revertir y aprender. También obligan a cerrar etapas. Si una tarea no puede describirse en una frase, quizá todavía es demasiado grande.

Ramas: probar sin romper

Una rama permite trabajar en cambios sin tocar la versión estable. Podés crear una rama para un lote de artículos, una corrección de diseño o una prueba técnica.

Cuando termina, se compara con la rama principal. Si está bien, se integra. Si no, se corrige sin haber destruido el sitio público.

Pull request: la mesa de revisión

Un pull request muestra qué cambió y permite revisar antes de mezclar. En este proyecto, la PR del lote de Tecnología funciona como zona de reconstrucción: los artículos se reescriben, se auditan y recién después pueden pasar a estable.

No es burocracia. Es protección.

Issues y notas

Las issues pueden servir para tareas, ideas, errores o decisiones pendientes. No necesitás un sistema enorme. Un título claro, una descripción y un estado ya ayudan.

También podés guardar archivos de avance dentro del repositorio: planes, manifiestos, auditorías y checklists.

GitHub Pages

GitHub Pages permite publicar sitios estáticos desde un repositorio. Para páginas HTML, blogs simples y documentación, eso puede ser suficiente.

La ventaja no es solo publicar gratis o fácil. La ventaja es que el sitio publicado nace desde archivos versionados. Si el cambio rompe algo, podés volver al historial.

Actions: controles automáticos

GitHub Actions puede ejecutar tareas cuando cambiás archivos o abrís una PR. En un blog, puede revisar contenido mínimo, enlaces, metadatos, sitemap y estructura.

Eso no reemplaza criterio editorial. Pero sí evita que se filtren errores repetidos o plantillas vacías.

Un flujo simple

  1. Crear rama para el cambio.
  2. Modificar pocos archivos.
  3. Hacer commit claro.
  4. Abrir o actualizar PR.
  5. Ejecutar auditorías.
  6. Revisar visualmente.
  7. Integrar solo si pasa.

Este flujo parece lento, pero acelera cuando el proyecto crece. El caos también tiene costo.

Qué guardar en el README

El README debería explicar qué es el proyecto, cómo se edita, cómo se publica, qué ramas importan, qué comandos se usan y qué no debe tocarse sin revisar.

Un README bueno evita que cada regreso al proyecto sea empezar de cero.

Errores comunes

  • Guardar cambios enormes sin mensaje útil.
  • Trabajar siempre directo en la rama estable.
  • No leer la diferencia antes de integrar.
  • Usar GitHub como depósito, pero no como taller.
  • No documentar cómo se publica.

GitHub ordena si lo usás con intención. Si solo subís archivos al azar, el desorden se muda de lugar.

Preguntas frecuentes

¿GitHub sirve si no soy programador experto?

Sí, especialmente para sitios estáticos, documentación y proyectos donde necesitás historial y control.

¿Qué es una rama?

Una línea de trabajo separada donde podés probar cambios sin afectar la versión estable.

¿Qué pongo en un commit?

Una unidad clara de cambio, con un mensaje que explique qué se hizo.

¿GitHub reemplaza mi copia local?

No. Ayuda mucho, pero conviene conservar también respaldos y saber recuperar archivos.
