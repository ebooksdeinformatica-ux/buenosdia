El proyecto no está ordenado si solo vos recordás dónde está todo

Organizar no es crear cincuenta carpetas. Es encontrar la versión vigente, entender qué falta y poder recuperar el trabajo cuando algo falla.

Por ASPF · Actualizado 12/06/2026 · 12–15 min
WhatsAppFacebookX

Perder un proyecto no siempre significa que el disco se rompió. A veces el archivo sigue ahí, pero nadie sabe cuál es el correcto. Hay tres carpetas con el mismo nombre, cinco versiones “finales”, imágenes mezcladas con descargas y una copia en la nube que sincronizó también el error.

Ese tipo de desorden no hace ruido hasta que necesitás volver. Entonces una corrección de diez minutos se convierte en una investigación arqueológica.

Organizar archivos no es decorar el escritorio. Es construir un sistema que responda sin depender de tu memoria:

  • ¿Dónde está la versión vigente?
  • ¿Qué material es original y qué material ya fue publicado?
  • ¿Qué cambió desde la última versión estable?
  • ¿Cuál es el próximo paso?
  • ¿Cómo recupero todo si esta computadora deja de funcionar?

Empezá por un proyecto, no por toda tu vida digital

Intentar ordenar años de archivos en una sola tarde suele terminar en otra carpeta llamada “ordenar después”. Elegí el proyecto que está activo o que más valor tiene. Puede ser una web, un catálogo, un libro, una campaña o un programa.

Creá una carpeta raíz con un nombre que no dependa del estado de ánimo:

buenosdia/

Adentro no necesitás veinte niveles. Una estructura corta puede alcanzar:

buenosdia/
├── 01-documentacion/
├── 02-contenido/
├── 03-imagenes/
├── 04-sitio/
├── 05-publicado/
└── 99-archivo/

Los números no son obligatorios, pero mantienen el orden visible. Lo importante es que cada carpeta tenga una función comprensible. Si necesitás abrirla para recordar qué contiene, el nombre no está ayudando.

Separá originales, trabajo y publicado

Uno de los errores más caros es modificar el único archivo que existe. Una imagen original, una versión optimizada y la imagen publicada no son lo mismo. Tampoco lo son las notas, el borrador y el HTML final.

En un proyecto web podrías separar:

  • Originales: fotografías, documentos recibidos y material sin editar.
  • Trabajo: archivos que todavía pueden cambiar.
  • Publicado: la versión exacta que llegó al sitio.
  • Archivo: versiones antiguas que ya no están activas pero todavía conservás.

Eso evita sobrescribir la única copia buena y permite comparar lo que ves online con lo que tenés guardado.

Nombres de archivo que expliquen algo

final-final-ahora-si.html parece gracioso hasta que hay que decidir si es más nuevo que final-corregido2.html.

Un nombre útil puede incluir proyecto, pieza, estado y fecha cuando la fecha aporte información:

buenosdia-portada-borrador.html
buenosdia-portada-revision.html
buenosdia-portada-publicada-2026-06-12.html

Para archivos gestionados con Git no hace falta meter un historial completo en el nombre, porque el sistema ya registra cambios. Para documentos comunes, una convención corta evita confusión.

Usá siempre el mismo formato de fecha, preferentemente AAAA-MM-DD. Así los archivos también se ordenan cronológicamente. Evitá símbolos extraños y nombres kilométricos si después se usarán en URLs o automatizaciones.

Git no es solamente para grandes programadores

El libro oficial de Git define el control de versiones como un sistema que registra cambios de archivos a lo largo del tiempo para poder recuperar versiones específicas. Puede utilizarse con código, pero también con archivos de texto, HTML, CSS, documentación y muchas otras piezas de un proyecto.

Git permite:

  • Ver qué cambió.
  • Volver a una versión anterior.
  • Trabajar en una rama sin romper la versión principal.
  • Comparar cambios.
  • Guardar mensajes que expliquen por qué se modificó algo.

Un repositorio de GitHub contiene archivos y su historial de revisiones. Las ramas son versiones paralelas y los pull requests permiten proponer cambios antes de aplicarlos a la rama principal. Esa lógica sirve perfectamente para un blog: escribís en una rama, revisás, auditás y recién después publicás.

No metas en Git todo lo que encontrás. Videos pesados, copias duplicadas, credenciales, archivos temporales y material privado requieren otro tratamiento. Un archivo .gitignore puede excluir elementos que no deben versionarse.

Git tampoco reemplaza un respaldo

Un historial de versiones ayuda a recuperar cambios, pero no resuelve todos los riesgos. Si el único repositorio está en una computadora que se rompe, sigue existiendo un problema. Si una cuenta es comprometida o borrás accidentalmente el remoto, necesitás otra copia.

Un respaldo razonable debería estar separado del original. Puede incluir:

  • La copia de trabajo en la computadora.
  • Un repositorio remoto o servicio confiable.
  • Una copia externa periódica que no permanezca conectada todo el tiempo.

La nube que sincroniza automáticamente puede ser útil, pero sincronizar no siempre equivale a respaldar. Si borrás un archivo o lo reemplaza una versión dañada, el cambio puede propagarse. Revisá si el servicio conserva historial y papelera, y probá recuperar un archivo.

El archivo más importante: LEEME

Cuando retomás un proyecto meses después, no necesitás solamente archivos. Necesitás contexto. Creá un README.md o LEEME.txt en la carpeta raíz con información básica:

  • Qué es el proyecto.
  • Dónde está publicado.
  • Qué tecnología utiliza.
  • Cómo se genera o publica.
  • Qué carpetas no deben tocarse.
  • Qué está pendiente.
  • Cuál fue la última decisión importante.

No pongas contraseñas dentro. Indicá dónde se administran de manera segura.

Este archivo transforma una carpeta muda en un proyecto que puede explicarse. También ayuda si en algún momento otra persona tiene que colaborar.

Cerrá cada sesión dejando una pista

La continuidad se rompe cuando cada día empezás preguntándote qué estabas haciendo. Al terminar una sesión:

  1. Guardá los archivos.
  2. Probá que la versión actual funcione.
  3. Hacé un commit o una copia identificable.
  4. Anotá el próximo paso concreto.
  5. Respaldá si hubo un avance importante.

“Seguir mejorando el sitio” no es un próximo paso. “Corregir el menú móvil en assets/css/main.css” sí.

Qué hacer con años de desorden

No intentes clasificar cada archivo viejo antes de empezar a producir. Creá una carpeta entrada-sin-clasificar, mové allí el material que todavía no revisaste y evitá que siga mezclándose con el trabajo activo.

Después procesá por prioridad:

  • Archivos únicos o irremplazables.
  • Proyectos que generan dinero o tienen valor estratégico.
  • Documentación legal o administrativa.
  • Material que se usa actualmente.
  • Duplicados y descargas prescindibles.

Eliminar también es organizar. Conservá lo que tiene una razón. Guardar cinco copias indistinguibles no es seguridad; es ruido.

Prueba de recuperación

Un respaldo que nunca probaste es una esperanza. Elegí un archivo, eliminá una copia de prueba y recuperalo desde el sistema de respaldo. Después comprobá que abre y que es la versión correcta.

Para un proyecto web, cloná el repositorio en otra carpeta o equipo y seguí el README. Si no podés reconstruir el sitio, falta documentación, falta una dependencia o algún archivo importante nunca entró al sistema.

Señales de que el sistema funciona

  • Encontrás la versión correcta en menos de un minuto.
  • Podés explicar el estado del proyecto sin abrir veinte carpetas.
  • Sabés qué archivo está publicado.
  • Podés volver atrás después de un cambio fallido.
  • Existe una copia fuera de la computadora principal.
  • El próximo paso queda escrito.

El objetivo no es tener el escritorio más lindo. Es recuperar energía mental. Cada vez que el sistema responde por vos, dejás de usar memoria para sostener cosas que una carpeta, un historial y una nota pueden sostener mejor.

Preguntas frecuentes

¿Conviene ordenar por fecha o por proyecto?

Para trabajo activo suele ser más claro ordenar primero por proyecto y usar fechas dentro de archivos o subcarpetas cuando aportan contexto. Una carpeta general por mes puede mezclar trabajos sin relación.

¿Git sirve para documentos de Word o imágenes?

Puede versionarlos, pero compara mejor archivos de texto. Para binarios grandes o material pesado conviene evaluar almacenamiento específico y respaldos separados.

¿Dropbox, Drive o OneDrive ya son un respaldo?

Pueden formar parte del sistema, especialmente si conservan historial, pero la sincronización también puede propagar borrados o archivos dañados. Conservá una copia separada y probá recuperación.

¿Cuántas carpetas debería tener un proyecto?

Las necesarias para distinguir funciones reales. Si la estructura obliga a recorrer ocho niveles para abrir un archivo cotidiano, probablemente está demasiado fragmentada.
